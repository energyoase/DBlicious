# 1.7.9 — Typst-PDF-Backend Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ein produktives PDF-Backend `TypstRenderer`, das hinter dem bestehenden `PdfRenderer`-Trait (`server/src/pdf/mod.rs`) echtes PDF aus einem Typst-Template + Variablen erzeugt — neben dem heutigen `StubRenderer`.

**Architecture:** `TypstRenderer` kompiliert eine Typst-Quelle in-process über `typst-as-lib` (wrappt den `World`-Trait), mit eingebetteten Default-Fonts aus `typst-assets`. Variablen werden **nicht** per String-Interpolation ins Template gespritzt (Injection-Risiko), sondern als JSON über `sys.inputs` übergeben; das Template liest sie mit `json(bytes(sys.inputs.at("data")))`. PDF-Export via `typst-pdf`.

**Tech Stack:** Rust, `server`-Crate, `typst` + `typst-as-lib` + `typst-assets` (embedded fonts) + `typst-pdf`, bestehender `PdfRenderer`-Trait, Tests über `%PDF-`-Magic + `pdf_extract` (dev-dep) für Text-Verifikation, `cargo test`.

**Roadmap-Bezug:** ROADMAP.md Zeile 1.7.9 (Status heute „🟡 Trait + Stub; Typst-Backend offen"). Akzeptanz: „typst preferred, fallback wkhtmltopdf". Dieser Plan liefert den typst-Pfad.

**Warum dieser Plan existiert (statt direkt umgesetzt):** Die typst-Rust-API ist zwischen Minor-Versionen instabil; `typst-as-lib` und `typst` 0.14 koppeln eng. Eine blinde Implementierung kostet viele Debug-Iterationen. Darum beginnt der Plan mit einem **verbindlichen Spike (Task 1)**, der die echte API festnagelt, bevor der Rest gebaut wird.

**Decisions aufgenommen:**
- **Crate-Wahl: `typst-as-lib` (primär)** statt direktem `typst` + eigenem `World`-Trait. Begründung: der `World`-Trait hat ~7 Methoden (library/book/main/source/file/font/today) plus Font-Management; `typst-as-lib` kapselt das. Falls der Spike (Task 1) zeigt, dass die API nicht trägt, ist der **dokumentierte Fallback** ein `TypstCliRenderer` (Subprocess `typst compile - --format pdf`), analog zum S3-`#[ignore]`-Smoke-Muster — siehe „Fallback-Pfad" unten.
- **Fonts: `typst-assets` mit `fonts`-Feature** — liefert die Typst-Default-Fonts (New Computer Modern u.a.) als `&'static [u8]`, kein eigenes TTF-Asset im Repo nötig.
- **Vars-Injection über `sys.inputs` (JSON), nicht String-Interpolation.** Verhindert Typst-Code-Injection durch Nutzer-Daten und erlaubt strukturierte Daten (Listen für Tabellen). Das `StubRenderer`-Verhalten (Template+Vars als Text einbetten) bleibt unverändert — die zwei Backends interpretieren `template`/`vars` bewusst unterschiedlich (Stub = Debug-Dump, Typst = echtes Typesetting).
- **`pdf_templates`-Tabelle + Designer-Anbindung bleiben Folge-Item** (Roadmap: „als Folge"). Dieser Plan nimmt `template: &str` direkt aus dem Trait-Argument; woher die Quelle kommt (Loader-Datei, DB-Tabelle, Designer) ist orthogonal.

## Architektur-Details

### `PdfRenderer`-Trait (existiert, unverändert)

```rust
// server/src/pdf/mod.rs (heute)
pub type PdfVars = serde_json::Map<String, serde_json::Value>;

#[async_trait]
pub trait PdfRenderer: Send + Sync {
    fn kind(&self) -> &'static str;
    async fn render(&self, template: &str, vars: &PdfVars) -> Result<Vec<u8>, PdfError>;
}

pub enum PdfError {
    TemplateInvalid(String),  // Compile-Fehler im Typst-Source
    Render(String),           // alles andere (Font-Init, PDF-Export, JSON-Encode)
}
```

`TypstRenderer` implementiert genau diesen Trait — Aufrufer (heute keine produktiven; künftig der `pdf_templates`-Pfad) wählen über `kind() == "typst"` vs `"stub"`.

### Vars-Bridge: `PdfVars` → `sys.inputs`

Das Template bekommt **eine** Input-Variable `data`, deren Wert der JSON-String der `PdfVars` ist. Im Template:

```typst
#let data = json(bytes(sys.inputs.at("data", default: "{}")))
= Rechnung #data.invoice_no
Kunde: #data.customer_name
```

Rust-seitig: `serde_json::to_string(vars)` → als `sys.inputs`-Eintrag `"data"` übergeben. So bleibt die Trait-Signatur (`vars: &PdfVars`) erhalten und es gibt keine String-Interpolation ins Quell-Template.

### Font-Lazy-Init

Fonts einmal prozessweit laden (sie sind `&'static [u8]` aus `typst-assets`), nicht pro `render`-Aufruf:

```rust
use std::sync::OnceLock;
fn fonts() -> &'static Vec<Vec<u8>> {
    static FONTS: OnceLock<Vec<Vec<u8>>> = OnceLock::new();
    FONTS.get_or_init(|| typst_assets::fonts().map(|f| f.to_vec()).collect())
}
```

(Exakte `typst_assets::fonts()`-Signatur in Task 1 verifizieren — Iterator-Item ist vermutlich `&'static [u8]`.)

### Fallback-Pfad (nur falls Task-1-Spike scheitert)

Wenn `typst-as-lib` 0.x die hier angenommene API nicht bietet und kein gangbarer Ersatz in <1h gefunden wird: statt In-Process ein `TypstCliRenderer`, der `typst compile - --format pdf --input data=<json>` als Subprocess aufruft (stdin = Template, stdout = PDF). Tests dann `#[ignore]` (braucht `typst`-Binary im PATH), analog `server/tests/storage_s3.rs`. Dieser Fallback ist **Plan B** — Task 1 entscheidet, ob er nötig ist. Nicht beide Pfade bauen.

### Task 3: `render()` implementieren + `PdfRenderer`-Impl

**Files:**
- Modify: `server/src/pdf/typst.rs`

- [ ] **Schritt 1: Render-Body**

An `server/src/pdf/typst.rs` anhängen (Signaturen ggf. an Task-1-Befund anpassen):

```rust
#[async_trait]
impl PdfRenderer for TypstRenderer {
    fn kind(&self) -> &'static str { "typst" }

    async fn render(&self, template: &str, vars: &PdfVars) -> Result<Vec<u8>, PdfError> {
        let data_json = serde_json::to_string(vars)
            .map_err(|e| PdfError::Render(format!("vars-encode: {e}")))?;

        // --- typst-as-lib (an Task-1-API anpassen) ---
        let engine = typst_as_lib::TypstEngine::builder()
            .main_file(template.to_string())
            .fonts(fonts().iter().map(|f| f.as_slice()))
            .build();

        let compiled = engine
            .compile_with_input("data", data_json);  // API ggf. .with_input(...)

        let document = compiled.output.map_err(|diags| {
            // typst-Diagnostics → erste Fehlermeldung extrahieren
            let msg = format!("{diags:?}");
            PdfError::TemplateInvalid(msg)
        })?;

        let pdf = typst_pdf::pdf(&document, &typst_pdf::PdfOptions::default())
            .map_err(|e| PdfError::Render(format!("pdf-export: {e:?}")))?;
        Ok(pdf)
    }
}
```

> **Hinweis an Implementierer:** Die exakten Methodennamen (`main_file`/`with_source`, `compile_with_input`/`compile`, `output`-Feld vs. `Result`) stammen aus dem Task-1-Spike. Wenn der Spike abweichende Namen ergab, hier konsequent ersetzen — die Struktur (build → compile-with-input → document → pdf-export) bleibt gleich.

use server::pdf::{PdfRenderer, TypstRenderer};
use serde_json::json;

fn vars(pairs: &[(&str, serde_json::Value)]) -> serde_json::Map<String, serde_json::Value> {
    pairs.iter().map(|(k, v)| (k.to_string(), v.clone())).collect()

async fn renders_valid_template_to_pdf_magic_bytes() {
    let tpl = std::fs::read_to_string("../examples/shop/pdf-templates/invoice.typ").unwrap();

    assert_eq!(&pdf[..5], b"%PDF-", "PDF-Magic erwartet");

#[tokio::test]
async fn invalid_template_yields_template_invalid_error() {

    assert!(

        "erwartete TemplateInvalid, war {err:?}"

}

#[tokio::test]
async fn rendered_pdf_contains_injected_vars() {
    let tpl = std::fs::read_to_string("../examples/shop/pdf-templates/invoice.typ").unwrap();
    let r = TypstRenderer::new();
    let v = vars(&[

    assert!(text.contains("INV-2026-000042"), "invoice_no fehlt im PDF-Text: {text}");
    assert!(text.contains("Mueller KG"), "customer_name fehlt: {text}");

- 1.7.9 „typst preferred" → Tasks 1-5 (typst-as-lib in-process) ✓
- „fallback wkhtmltopdf" → ersetzt durch dokumentierten `TypstCliRenderer`-Fallback (Task 1 Gate); wkhtmltopdf ist EOL, CLI-Subprocess ist der pragmatische Ersatz ✓

## Risiken / Hinweise

- **typst-API-Drift:** Höchstes Risiko. Task 1 ist das Mitigation-Gate. Bei Scheitern → CLI-Fallback.
- **Build-Größe:** typst + typst-assets-fonts ziehen ~30-50 MB. Bewusst akzeptiert (Roadmap-Vorbehalt). Nur im `server`-Crate, kein WASM-Impact.
- **Modulname `typst`:** kollidiert mit dem Crate. Falls Rust die Auflösung nicht sauber trennt (`::typst` vs `crate::pdf::typst`), Modul in `typst_backend` umbenennen — Entscheidung in Task 1, dann konsistent durch alle Tasks ziehen.
- **Multi-Session-Vorsicht:** Solange parallele Sessions im selben Repo committen, jeden Commit mit explizitem Pathspec (`-- <pfade>`) machen; `server/src/pdf/stub.rs` und andere fremde uncommitted Dateien nicht stagen.
